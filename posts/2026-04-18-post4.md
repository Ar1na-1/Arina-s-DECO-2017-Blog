# Designing the System Structure and Data Model

## Problem Context

While interaction design defines how users engage with a system, the underlying structure determines whether those interactions can be effectively supported. A community platform that enables structured feedback must rely on a clear data model to maintain relationships between posts, users, and feedback.

Without a well-defined structure, it becomes difficult to organise, retrieve, and present feedback in a meaningful way.

## System Overview

The system is designed around three core entities:

- Users
- Posts
- Feedback

Each entity represents a fundamental component of the community interaction, and their relationships define how information flows within the system.

## Data Structure

### Users

Each user has a unique identity, which is used to associate posts and feedback.

- user_id  
- username  

Users can both create posts and provide feedback, reflecting the dual role of participants within the community.

### Posts

A post represents a piece of content shared by a user, along with a defined feedback goal.

- post_id  
- user_id (author)  
- title  
- description  
- feedback_goal  

The inclusion of a feedback goal is critical, as it provides context for interpreting responses.

### Feedback

Feedback is structured and linked to both a post and a user.

- feedback_id  
- post_id  
- user_id (reviewer)  
- clarity  
- confusion  
- suggestion  

This structure ensures that feedback remains organised and aligned with the original post.

## System Relationships

The system follows a relational model:

- One user can create multiple posts  
- One post can have multiple feedback entries  
- Each feedback entry belongs to one post and one user  

This structure allows the system to efficiently retrieve all feedback associated with a post and present it in a structured format.

## Design Decisions

The decision to separate feedback into structured fields (clarity, confusion, suggestion) directly supports the goal of improving relevance and actionability.

Rather than storing feedback as a single block of text, this approach enables the system to categorise and display responses more clearly.

It also creates opportunities for future features, such as filtering or aggregating feedback by type.

## Alternative Approaches

An alternative would be to store feedback as unstructured text. While this simplifies the data model, it makes it more difficult to organise and interpret responses.

Another approach would involve a more complex schema, such as supporting multiple feedback categories or scoring systems. While this could increase analytical depth, it would also increase implementation complexity and user effort.

The chosen model prioritises simplicity while still supporting structured interaction.

## Trade-offs

The simplified data model is easy to implement and understand, but it limits flexibility. Users must conform to predefined categories, which may not capture all types of feedback.

Additionally, storing structured fields requires more deliberate input from users, which may reduce participation.

However, these trade-offs are acceptable, as the system prioritises clarity and usefulness over unrestricted input.

## Conclusion

A well-defined system structure is essential for supporting structured, goal-oriented feedback in a community platform. By organising data into clear entities and relationships, the system enables consistent, interpretable, and actionable feedback.
