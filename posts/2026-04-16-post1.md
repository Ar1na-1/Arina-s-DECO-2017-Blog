# Defining Functional Requirements for a Context-Aware Usability Feedback Community

## 1. Problem Context

In design education, usability testing is a critical step in evaluating the effectiveness of an interface. However, for many students, conducting meaningful usability testing is difficult due to limited access to real users and structured evaluation methods.

As a result, students often rely on informal peer feedback, which tends to be unstructured, subjective, and difficult to translate into actionable improvements.

---

## 2. Problem Analysis

The core issue is not the lack of feedback, but the lack of context in which feedback is given. When reviewers do not understand the designer’s specific goal, feedback often becomes irrelevant or superficial.

For example, a designer aiming to test navigation clarity may receive comments on visual style, which does not contribute to the intended evaluation.

This mismatch reduces the usefulness of feedback and limits its role in supporting iterative design.

---

## 3. Proposed Solution

To address this issue, this project proposes a lightweight community hub that enables design students to conduct rapid usability testing through structured peer feedback.

The key idea is to introduce context-aware feedback by requiring users to define a usability goal when posting their work, and guiding reviewers to respond in a structured format aligned with that goal.

This transforms feedback from a free-form activity into a guided usability evaluation process.

---

## 4. Functional Requirements

### Core Functional Requirements

- Users must be able to create a post that includes a defined usability testing goal.  
- The system must present posts along with their associated testing goals.  
- Users must provide structured feedback based on predefined prompts (e.g. clarity, confusion, suggestions).  
- The system must display feedback in relation to the original usability goal.  

---

### Optional Features

- Filtering feedback by category  
- Supporting different stages of design (e.g. early vs final)  

---

### Scope Decisions

At this stage, features such as user reputation systems, expert matching, or automated analysis are intentionally excluded to maintain a focused and achievable scope.

---

## 5. Alternative Approaches Considered

An alternative approach would be to allow completely free-form feedback without any structure. While this would reduce friction and make participation easier, it would likely result in low-quality or irrelevant responses, as reviewers may not align their feedback with the designer’s intended goal.

Another possible approach would be to implement automated feedback systems, such as AI-generated usability suggestions. However, this introduces technical complexity and may reduce the authenticity and diversity of peer feedback.

Therefore, the decision to use structured, goal-oriented peer feedback represents a balance between usability, feasibility, and meaningful interaction.

---

## 6. Trade-offs and Constraints

Introducing structured feedback improves relevance and actionability, but reduces flexibility in how users express their opinions.

Similarly, requiring users to define a usability goal increases the quality of responses, but adds friction to the posting process.

These trade-offs are considered acceptable, as the primary goal of the system is to prioritise meaningful usability insights over casual interaction.

---

## 7. Technical Alignment

The decision to use structured input fields instead of free-form comments directly supports the functional requirement of improving feedback relevance. By constraining user input into predefined categories, the system ensures that feedback remains aligned with the usability goal defined by the post author.

This design also simplifies data handling and presentation, as structured responses can be more easily grouped, compared, and analysed within the interface.

---

## 8. Evaluation and Responsibility

To evaluate the effectiveness of this system, usability testing can be conducted to assess whether users are able to provide more relevant and actionable feedback compared to unstructured systems. Metrics such as feedback clarity, alignment with the stated goal, and user satisfaction can be used.

Accessibility considerations include ensuring that all form elements are properly labelled, keyboard navigation is supported, and visual hierarchy is clear. Additionally, the system should encourage respectful and constructive feedback to minimise potential negative or harmful interactions within the community.

These considerations ensure that the system is not only functional, but also responsible and inclusive.

---

## 9. Conclusion

This project focuses on improving the quality of peer feedback in design communities by embedding usability testing principles into the interaction process.

By prioritising structured, goal-oriented feedback, the system aims to support more effective and iterative design practices among students.
